# JpoProducer SPEC v1.0（凍結仕様）

**最終更新:** 2026-07-15  
**位置づけ:** プロダクトの**唯一の真実**。README / HANDOVER / エージェント指示はこれに従う。  
**版:** v1.0（Ship 目標）／ v1.1+ は本文末尾のみ。

---

## 1. プロダクト一文

> **J-Pop / J-Rock 特有の「細かく・速いコード切り替え」とシンコペを、まず最速で置き、  
> その上にメロディが浮かぶ程度の単純伴奏を一瞬で敷き、  
> 普通の MIDI 編集で整え、4/8/16 小節ループをつないで曲の骨格にし、  
> 必要なら Grok に進行を渡してパート（ストリングス等）を足す。**

Domino / DAW の置き換えではない。**進行と骨格の爆速スケッチ**が本体。

---

## 2. 五本柱（これ以外は v1 に入れない）

| # | 柱 | ユーザー価値 | 市場との差 |
|---|-----|-------------|-----------|
| **P1** | **密なコード進行** | 1 小節に複数コード、ハーフ拍・16 分、前ノリ（シンコ）をストレスなく | 多くは「1 小節 or 1/2 小節 1 コード」が得意で、それより細かいと遅い |
| **P2** | **単純伴奏ベッド** | 進行に対し Piano+Bass+Drum の薄いベッドをサクッと | アレンジ詰めはしない。メロが書きやすい土台だけ |
| **P3** | **普通の MIDI 編集** | 選択・描画・消・移動・伸縮・コピペ・Undo・再生 | Tab3 が目指していた Domino 級の「普通」 |
| **P4** | **ループ接続** | 4 / 8 / 16 小節ループを Bank に貯め、並べて曲の骨格 | 無限タイムラインにしない |
| **P5** | **Grok パート生成** | 進行＋区間を Grok に渡し、「この 4 小節にストリングス」等の MIDI を取り込む | 進行専用ツールや DAW 単体にない共創 |

## 3. 情報アーキテクチャ（UI 提案）

### 3.1 なぜ「後付け 4 タブ」は中途半端か

現行 4 タブ（Chord / Generate / Edit / Arrange）は、**ショートカット衝突と状態バグを隔離する**ための応急処置として正しかった。  
しかし創作フローは本来こう動く:

```text
密な進行を書く → ベッドを敷く → メロ/パートを書く・Grok で足す → ループを並べる
```

Generate と Edit を完全分離すると、「伴奏を出してすぐ直す」「Grok で足してすぐ直す」たびにタブ往復が発生する。  
Grok が下部パネルに埋まっているのも、**P5 を副機能に見せてしまう**。

### 3.2 推奨ターゲット UX: **3 ワークスペース**

使いやすさ優先。タブ数そのものより **「今何をしているか」** が一つの画面に収まること。

| Workspace | 役割 | 主な操作 | やらないこと |
|-----------|------|----------|--------------|
| **1 Progress** | コード進行専用エディタ | 細粒度配置・伸縮・移動・パレット・シンコ印・J-Pop スタンプ | ノート編集、伴奏生成の詳細 |
| **2 Sketch** | 1 ループの制作机 | ベッド生成、ピアノロール編集、Grok パート、再生 | 曲全体の並べ替え |
| **3 Arrange** | 曲の骨格 | Loop Bank、順序、通し再生、フル MIDI export | ノート編集 |

```text
┌──────────── Progress ────────────┐
│  CHORD TIMELINE（高解像度グリッド） │
│  パレット / スタンプ / シンコ        │
└──────────────────────────────────┘

┌──────────── Sketch ──────────────┐
│ onion 進行（薄） │ トラック │ ツール │
│ ──── ピアノロール（編集 SoT）──── │
│ [Simple Bed] [Clear Bed]  パターン任意 │
│ ┌ Grok Parts ─────────────────┐  │
│ │ 区間・トラック・プロンプト     │  │
│ │ Copy context → paste/import  │  │
│ └─────────────────────────────┘  │
└──────────────────────────────────┘

┌──────────── Arrange ─────────────┐
│ Loop Bank │ スロット列 │ 通し再生  │
└──────────────────────────────────┘
```

**入力隔離（バグ再発防止・タブ数に依存しない）:**

| フォーカス領域 | 有効な操作 |
|----------------|------------|
| Chord strip（Progress、または Sketch 上の進行帯） | ブロック操作のみ。Ctrl+C/V はブロック用 or 無効 |
| Piano roll（Sketch） | ノート Select/Draw/Erase、Ctrl+C/X/V/D、Undo |
| Grok dock | テキスト・import。ロールのショートカットを奪わない |
| Arrange | Bank/スロットのみ |

実装では「タブ」でも「1 画面内のフォーカス」でもよい。**隔離ルールを守ることが不変条件**。

### 3.3 現行コードとの対応（移行）

| 現行 | ターゲット | 移行方針 |
|------|------------|----------|
| Tab1 Chord | Progress | ほぼそのまま強化（密グリッド・スタンプ） |
| Tab2 Generate | Sketch の「Bed」ツール群 | **画面統合が理想**。v1.0 までは Tab2 残置可 |
| Tab3 Edit | Sketch のピアノロール | 本体 |
| 下部 Grok import | Sketch の **Grok Parts ドック** | 格上げ（必須） |
| Tab4 Arrange | Arrange | そのまま |

**v1.0 Ship 条件:** 五本柱が手で回ること。3 ワークスペースへの見た目統合は **v1.0 推奨だが、4 タブのまま柱を満たせば Ship 可**。  
統合は「挙動を変えずにパネル配置を寄せる」作業とし、生成アルゴリズムと同時にやらない。

### 3.4 代替案（不採用だが記録）

| 案 | 内容 | 不採用理由 |
|----|------|------------|
| 5 タブ（Grok 独立） | Progress / Bed / Edit / Grok / Arrange | Grok は「生成して Edit に載せる」流れなので独立タブは往復が増える |
| 完全 1 画面復活 | 旧 v0 型 | ショートカット事故の再来リスク。フォーカス隔離なしでは戻さない |
| Domino クローン | フルイベントリスト等 | スコープ外。P3 は「普通のピアノロール」まで |

## 4. 柱ごとの仕様

### 4.1 P1 — 密な J-Pop コード進行

**データ（SoT）:** `ChordBlock { start, dur, degree, quality, octave, syncopation_fill }` のリスト。  
Ch1 の実 MIDI ノートが真実ではない（export 時に展開）。

**時間解像度（v1 必須）:**

| 項目 | 仕様 |
|------|------|
| 配置グリッド | 既定 **1/16 拍**（必要なら 1/8・1/12 三連を選択可） |
| 最小ブロック長 | **0.25 拍（16 分）** |
| 1 小節内コード数 | **制限なし**（UI が読みやすければ 4〜8 個/小節も可） |
| シンコ / 前ノリ | ブロック `start` を拍頭より前に置ける。`syncopation_fill` は伴奏差し替えヒント |
| 非重複 | `enforce_chord_timeline_no_overlap`（**ブロック削除で解決しない**） |
| 操作 | 空きクリック配置、右端伸長、ドラッグ移動、パレットで quality |

**J-Pop 向け UI（Progress 強化・優先順）:**

1. **高コントラストの小節線 + 拍線 + 16 分薄線**（密配置でも位置が読める）
2. **短いブロックでもラベルが読める**（省略名 `Em` / `G◆` 等、ホバーで全文）
3. **進行スタンプ（プリセット）** — 進行の**末尾に追記**（空なら 0）。playhead 非依存。  
   作り直し・比較は進行クリア → 別スタンプ。UI はコンボ + 追記（全部表出しない）。  
   - 例: 王道 I–V–vi–IV（1 小節ずつ）  
   - 例: **半分割り** I–V | vi–IV  
   - 例: **16 分詰めデモ**（密配置のチュートリアル用）  
4. **前ノリ補助（任意・v1.0 できれば）** — 選択ブロックを −1/8 or −1/16 シフト

**やらない (v1):** 自動転調解析、リアルタイムコード認識、ローマ数字以外の複雑な理論 UI。

### 4.2 P2 — 単純伴奏ベッド

**目的:** メロディが浮かぶ土台。アレンジ作品にしない。

**v1 の主操作:**

| 操作 | 挙動 |
|------|------|
| **Simple Bed** | 現在ループ（または gen 範囲）に Piano(Ch2)+Bass(Ch3)+Drum(Ch10) を一括生成 |
| **Clear Bed** | Ch2/3/10 の該当範囲をクリア |
| パターン選択 | 任意。既定は「一番素直な」Piano / Bass / Drum 各 1 種 |
| Preview | プロジェクトを壊さず試聴（現行 Preview 相当）→ 気に入ったら Apply/Generate |

**生成の音楽契約:**

```text
pattern MIDI → 拍グリッド配置（BPM 非依存）
  → 音程マップ（Piano: chord quality / Bass: register E1–D#2 / Drum: 無変換）
  → cleanup（ユニーク NoteId、same-pitch 時間 overlap 解消、Bass 帯域）
  → replace_notes_in_range
```

| 現象 | 扱い |
|------|------|
| quality 無視（Em→G#） | **バグ**（修正済み・回帰必須） |
| 同一 pitch の時間重なり | **バグ**（cleanup・回帰必須） |
| 和音の同時発音（異 pitch） | **仕様**（ポリフォニー） |
| テンプレ Gate のべったり感 | **assets 品質**（Domino で短縮。コードで全和音禁止はしない） |
| 高度なボイシング・カウンターメロ | **対象外**（手編集 or Grok パート） |

**実装手段:** 現行のパターン MIDI 方式でよい。  
「最も簡単で効率的」が後からアルゴリズム生成に変わっても、**契約（上表）と Simple Bed UX を保てば差し替え可**。

### 4.3 P3 — 普通の MIDI 編集（Sketch ピアノロール）

**対象:** Ch2–16（Ch1 はコード SoT。ロール上の Ch1 は読み取り推奨）

| 必須 | 内容 |
|------|------|
| ツール | Select / Draw / Erase（排他） |
| 選択 | クリック、マーキー、Ctrl+複選。ハイライトは **選択 id のみ** |
| 編集 | 移動、右端 Gate、Delete |
| クリップボード | Ctrl+C/X/V/D。**貼付基準 = playhead** |
| playhead | タイムラインクリックで挿入位置 |
| Snap | 既定 ON。位置と Gate は概念上分離（実装は段階可） |
| Undo/Redo | プロジェクトスナップショットで可（深度は浅くてよい） |
| オニオン | スケール + コードトーン（濃さ調整） |
| 再生 | Space、SF2、ループ長 4/8/16。**Transport（▶）は常時表示・スクロールで隠れない** |
| 縦表示 | 既定 **約 2 オクターブ**（手書き密度）。ズームで拡大可 |
| 下レーン | **切替ツール帯**（Vel 棒グラフ必須。Grok 取込を同帯に置く）。将来 Sustain 等 |
| Grok 書込 | 文脈コピー + **MIDI / ノート列を選択トラック・playhead へ import**（NoteId 採番） |

**やらない (v1):** CC マクロ、ストローク、フルイベントリスト、プラグインホスト、Program Change の常時 UI。

### 4.4 P4 — ループ接続（Arrange）

| 項目 | 仕様 |
|------|------|
| ループ長 | **4 / 8 / 16 小節のみ**（自由長は作らない） |
| Loop Bank | 名前付き保存・複製 |
| シーケンス | スロット列 + 繰返し、↑↓ 程度で順序変更 |
| 再生 | 通し再生（Arrange タイムライン） |
| Export | シーケンス全体の Type 1 MIDI |

**やらない (v1):** 高度なドラッグ並べ替え UI、オートメーション、テンポマップ複数。

### 4.5 P5 — Grok パート生成（第一級機能）

**思想:** 進行はアプリが正確に持つ。Grok は **その進行に沿った MIDI パーツ** を返す共創相手。

**標準フロー:**

```text
1. ユーザーが Progress / Sketch で進行と区間を確定
2. [Copy Grok context] — Key/BPM/ループ長/コードタイムライン/指定トラック/指示文をクリップボードへ
3. ユーザーが Grok に貼り、MIDI または進行テキストを得る
4. [Import] — .mid を選択トラック・playhead（または指定区間）へ。NoteId 採番必須
5. Sketch で通常編集
```

**コンテキストに必ず含めるもの:**

- Key / Major-Minor / BPM  
- 対象拍範囲（例 0–16）  
- コードブロック一覧（start, dur, 表示名）  
- ユーザー指示テンプレ例: 「Ch4 にストリングスのパッドを 4 小節」「サビ用の単純カウンターメロ」

**v1 でやること / やらないこと:**

| やる | やらない |
|------|----------|
| コンテキストコピーの品質 | アプリ内から Grok API を直接叩く（認証・課金・オフラインを複雑化） |
| MIDI import を playhead / トラック指定で安定化 | Grok 出力の自動品質保証 |
| 進行 NL パース（既存の C\|Am\|F\|G 等）の維持 | 「全自動で名曲を吐く」 |

将来（v1.1+）: API 直結、構造化 JSON 進行の往復、パート種別プリセットボタン。

## 5. 画面スケール制約

| 項目 | v1 固定 |
|------|---------|
| ループ長 | 4 / 8 / 16 小節のみ |
| 既定表示 | **8 小節フィット** |
| 横方向 | 原則ループ内。パズル感を守る |
| Progress グリッド | 密配置可读（16 分線） |
| Piano 縦窓 既定 | おおむね C2–C7、スクロール可 |
| Bass 意識帯域 | 生成 E1–D#2（MIDI 28–51） |
| 最小ウィンドウ目安 | 幅 ≥ 1100、高 ≥ 700 |
| Snap | 既定 ON |

## 6. データ不変条件（要約）

詳細は `.grok/skills/jpo-producer/references/invariants.md`。SPEC として拘束する要点:

1. 編集モデルは `{id, start, dur, pitch, vel}`。SMF は I/O 境界のみ  
2. **NoteId はユニーク**（生成・import・paste で `NoteId(0)` 量産禁止）  
3. Ch1=進行 SoT、Ch2=Piano、Ch3=Bass、Ch10=Drum（転調なし）  
4. 時間は **拍グリッド・BPM 非依存**  
5. Bass 生成は 28–51  
6. Piano 生成は quality 反映 + same-pitch overlap 0  
7. コードブロック非重複（enforce で削除しない）  
8. `replace_notes_in_range` は範囲外を消さない

## 7. v1.0 Definition of Done（完成の定義）

次をすべて満たしたら **v1.0 完成** とし、機能追加を止めてタグを打つ。

### 自動

- [ ] `cargo test` 全パス（id / bass 帯域 / piano quality / same-pitch overlap / sync 窓 / chord enforce 等）

### 手動（`tests/golden/ACCEPTANCE.md`）

- [ ] **P1** 1 小節に 3 つ以上の短いコードを 16 分グリッドで置ける。シンコ位置が聴感・見た目で分かる  
- [ ] **P2** Simple Bed（または現行 Generate All）で P+B+D が載り、Em 等の quality が破綻しない  
- [ ] **P3** 1 ノート選択がそれだけハイライト。playhead へペースト。Undo が最低 1 手戻る  
- [ ] **P4** 2 つ以上のループを並べ、通し再生とフル MIDI export  
- [ ] **P5** Grok context をコピーし、戻った MIDI を指定トラックに import できる  
- [ ] `.jpo` 保存/再読込、Space 再生（SF2）

### 出荷

- [ ] version `1.0.0`  
- [ ] `pack.ps1` ポータブル zip（exe + SF2）が別 PC で起動・再生  

### 明示的に v1.0 に含めない

- Domino 全機能、API 直結 Grok、Arrange 本格 DnD、日本語フォント同梱必須、dim/sus 全品質の耳チューニング、`main.rs` 分割完了（**推奨だが DoD 外** — 分割は完成前保険として別タスク）

## 8. 実装ロードマップ（SPEC 準拠）

順序を守る。柱をまたぐ大きな変更を同時にしない。

| Step | 内容 | 柱 |
|------|------|-----|

## 9. プロセス規則

1. 仕様変更は **先に SPEC を更新**してからコード  
2. 1 セッション 1 テーマ  
3. 生成・pitch・cleanup 変更はテスト必須  
4. バグに合わせて golden を改ざんしない  
5. フルリライト禁止（既存テストが契約を守れる限り）  
6. Domino は手編集・参考のみ（GUI 自動化しない）  
7. 開発: WSL `~/JpoProducer` / 配布: Windows
